## User Scenarios & Testing

### User Story 1 - 网页端扫码登录咸鱼账号 (Priority: P1)

管理员在系统中添加咸鱼账号时，可以直接在网页前端完成登录授权，无需手动复制Cookie。

**Why this priority**: 这是核心功能，用户当前需要手动复制Cookie非常繁琐，必须实现网页端登录

**Independent Test**: 管理员点击"添加账号" → 系统显示咸鱼登录页面或二维码 → 管理员扫码登录 → 系统自动获取Cookie并保存 → 账号状态显示"在线"

**Acceptance Scenarios**:

1. **Given** 管理员进入账号管理页面, **When** 点击"添加账号"按钮, **Then** 系统弹出登录对话框，显示咸鱼登录页面或二维码
2. **Given** 登录对话框显示二维码, **When** 管理员使用闲鱼APP扫码并在手机上确认登录, **Then** 系统自动检测到登录成功，获取Cookie，保存到数据库，账号状态变为"在线"
3. **Given** 登录对话框显示账号密码登录界面, **When** 管理员输入咸鱼账号密码完成登录, **Then** 系统自动获取Cookie，保存到数据库，账号状态变为"在线"
4. **Given** 登录过程中出现错误, **When** 用户名密码错误或二维码超时, **Then** 系统显示错误提示，管理员可以重试

---

### User Story 2 - 登录状态自动检测与保持 (Priority: P1)

系统定期检测已登录账号的状态，在登录过期前主动提醒或自动重新登录，保持账号持续在线。

**Why this priority**: 账号登录状态需要持续维护，否则一段时间后Cookie会失效，影响自动发货等功能

**Independent Test**: 定时任务执行 → 检测所有账号Cookie有效性 → 发现失效账号 → 发送提醒或自动重新登录

**Acceptance Scenarios**:

1. **Given** 系统中有多个在线账号, **When** 定时任务每6小时执行一次登录状态检测, **Then** 系统使用各账号Cookie访问咸鱼API，验证Cookie有效性
2. **Given** 检测到某账号Cookie已失效, **When** 系统检测到登录状态为"已过期", **Then** 系统记录告警，发送邮件通知管理员，并标记账号状态为"已过期"
3. **Given** 账号状态为"已过期", **When** 管理员再次点击"扫码登录"重新授权, **Then** 系统获取新Cookie，更新数据库，账号状态恢复为"在线"
4. **Given** 系统配置了自动重新登录, **When** 检测到Cookie即将过期（如7天内）, **Then** 系统自动发起重新登录流程，发送通知提醒管理员扫码

---

### User Story 3 - 多账号管理与状态展示 (Priority: P2)

管理员可以清晰查看所有咸鱼账号的登录状态，方便统一管理。

**Why this priority**: 需要让管理员一目了然地知道哪些账号在线、哪些过期

**Independent Test**: 管理员查看账号列表 → 看到每个账号的实时状态 → 可以对每个账号进行操作

**Acceptance Scenarios**:

1. **Given** 账号列表页面, **When** 页面加载, **Then** 显示所有账号及其当前状态（在线/离线/过期）
2. **Given** 账号状态为"在线", **When** 管理员点击"刷新状态"按钮, **Then** 系统立即检测登录状态，更新显示
3. **Given** 账号状态为"过期", **When** 管理员点击"重新登录"按钮, **Then** 系统弹出登录对话框，引导管理员重新授权

---

### Edge Cases

- 用户扫码后长时间未确认登录，二维码超时如何处理？
- 咸鱼登录接口返回异常或频繁检测被限制如何处理？
- 同时有多个账号进行登录操作是否会冲突？
- 浏览器端登录后服务器端获取Cookie的时机如何把握？

## Requirements

### Functional Requirements

- **FR-001**: 系统必须在网页前端显示咸鱼登录页面或二维码，供用户扫码或账号密码登录
- **FR-002**: 系统必须在用户完成登录后自动获取Cookie，无需用户手动复制
- **FR-003**: 系统必须将获取的Cookie加密存储到数据库，保障安全性
- **FR-004**: 系统必须定期（建议每6小时）自动检测所有账号的登录状态
- **FR-005**: 系统必须在检测到登录过期时发送邮件通知管理员
- **FR-006**: 系统必须在账号列表页面清晰展示每个账号的登录状态
- **FR-007**: 系统必须支持管理员手动触发账号状态刷新
- **FR-008**: 系统必须支持登录超时、失败等异常情况的错误提示和重试

### Key Entities

- **账号 (Account)**: 咸鱼账号，包含昵称、加密的Cookie、登录状态、最后登录时间
- **登录会话 (LoginSession)**: 记录每次登录的时间，操作、结果
- **告警 (Alert)**: 登录状态异常时的通知记录

## Success Criteria

### Measurable Outcomes

- **SC-001**: 管理员可以在3分钟内完成整个扫码登录流程（从点击添加账号到账号显示在线）
- **SC-002**: 90%以上的账号在登录有效期内保持在线状态
- **SC-003**: 登录过期检测任务执行成功率不低于95%
- **SC-004**: 管理员能够清晰区分所有账号的当前状态（在线/离线/过期）
- **SC-005**: 从检测到登录过期到发送邮件通知的延迟不超过5分钟
